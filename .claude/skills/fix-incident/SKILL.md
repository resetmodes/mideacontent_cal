# 사고 대응 스킬 — 진단 → 복구 → 재발 방지 (복구만 하고 끝내면 반쪽)

## 1. 먼저 분류하라 (증상이 같아도 원인 층이 다르다)

| 층 | 예 | 첫 확인 |
|---|---|---|
| 데이터 | 계정·일정이 사라짐, 지표가 빔 | `git log --oneline -- <데이터파일>` 로 언제 바뀌었나 |
| 코드 | 기능이 동작 안 함 | `npm run verify` + `npm run smoke` 재현 |
| 배포 | 사이트에 반영 안 됨 | main에 커밋이 있나, Vercel 빌드(1~2분) 기다렸나 |
| 외부 | 수집 실패, 403 | GitHub Actions 로그 → Apify 한도/토큰 |
| 설정 | 특정 기능만 서버 오류 | 대기 SQL 미적용? (PLAYBOOK 4장 목록) |

**"코드 문제처럼 보이는 것의 상당수는 데이터·설정 문제다.** 코드를 고치기 전에
데이터 파일의 git 이력과 Supabase SQL 적용 여부부터 본다."

## 2. 데이터 복구 절차 (실제 '26.7.7 사고 2건에서 검증된 방법)

1. 마지막 정상 시점 찾기: `git log --oneline -- src/data/...` → 의심 커밋 전후를
   `git show <sha>:<파일> | grep ...` 로 비교 (계정 수, 배열 길이 등 정량 비교)
2. 복구는 **통째 되돌리기보다 병합**: 사고 커밋에도 신규 데이터(새 계정, posts 등)가
   섞여 있을 수 있다 — 이전 버전에서 사라진 항목만 뽑아 현재 파일에 합치는
   스크립트를 짜서 복구 (`'26.7.7 인스타 26계정 복구' 커밋 참고)
3. 캘린더(DB) 데이터는 `data/backup/media-events.json` (주 1회 자동 백업)의
   git 이력에서 시점 선택 → Supabase에 insert로 복원

## 3. Actions·수집 관련 철칙

- **수집 워크플로우 재실행 금지가 기본** — 실행마다 Apify 과금(~$1.7). 실패 원인이
  한도 초과(403 actor-disabled)면 재실행해도 또 실패 + 과금만 발생
- 로그 보는 법: Actions → 해당 run → collect job → "수집 + 정제" 스텝. 4단계(수집)
  성공 + 5단계(push) 실패면 과금은 됐고 데이터만 없는 상태
- push 충돌은 워크플로우가 rebase 3회 재시도로 자체 처리함

## 4. 재발 방지 3종 세트 (복구 후 반드시 — 이게 이 스킬의 핵심)

1. **가드 코드**: 같은 원인이 다시 와도 데이터가 안 상하게 (빈 결과 저장 스킵,
   carry-forward 같은 패턴 — clean-*.mjs 참고)
2. **회귀 테스트**: 사고 시나리오를 scripts/test-guards.mjs 에 추가 — 미래의 누가
   가드를 지우면 테스트가 잡게
3. **CLAUDE.md 사고 기록**: 언제·무엇이·왜 났고 어떻게 막았는지 한 항목 —
   다음 세션·다른 모델이 같은 함정에 안 빠지게

## 5. 보고

원인(한 문장) → 복구한 것 → 재발 방지로 심은 것 → 남은 위험. 추측이면 추측이라고 말할 것.
