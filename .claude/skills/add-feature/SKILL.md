# 기능 추가 스킬 — 설계 사고 → 구현 → 검증 → 배포

## 1. 코딩 전에 반드시 답할 3가지 (건너뛰면 재작업 난다)

1. **이 요청의 실제 문제가 뭔가?** 사용자가 말한 해법이 아니라 겪는 불편을 한 문장으로.
   확신이 없으면 구현 전에 물어볼 것 — 사용자(규빈)는 결론 선행형이고 임의 재구성을 싫어함.
2. **비슷한 패턴이 이 리포 어디에 이미 있나?** 새로 발명하지 말고 기존 패턴을 찾아 재사용:
   - 모달 안 2차 액션 → `md-spec-link` (스펙 딥링크) 방식
   - 조건부 섹션 → "값 있을 때만 노출" (메모, 실적 후보)
   - 새 저장 필드 → `kind`/`perfUrl` 패턴 (store.js 조건부 전송)
   - 자동 수집·기록 → DB 트리거(이력) 또는 GitHub Actions(백업·수집)
   - 파서 확장 → parse.js 라벨 추출(촬영/업로드) 방식
3. **안 만들어도 되는 부분은?** 최소로 시작. "없으면 숨김"이 이 리포의 기본값.

## 2. 이 리포의 확립된 설계 원칙 (전부 실사고·실피드백에서 나옴)

- **비침습**: 기존 클릭 동작(셀 클릭, 일정 클릭)은 절대 바꾸지 않는다. 새 동작은
  분리된 텍스트 링크·작은 버튼으로. 미스클릭 유발 구조는 사용자가 명시적으로 거부함
- **없으면 숨김**: 데이터·매핑·설정이 없으면 그 UI는 아예 안 보이게 (막다른 길 금지,
  에러 대신 숨김). 단 설정 미비 안내가 필요한 곳은 짧은 문구로
- **스키마 하위호환**: 새 DB 필드는 값이 있을 때만 전송 (`...(e.x ? {x: e.x} : {})`) —
  사용자가 SQL을 아직 안 돌렸어도 기존 기능이 절대 깨지지 않게. SQL은
  data/supabase-setup.md에 장을 추가하고 사용자 액션으로 안내
- **3뷰 확인**: 모든 UI 변경은 ①팀용 기본 ②?view=mirror·미러 사이트(읽기 전용)
  ③?view=external(외부, 캘린더 숨김) 에서 어떻게 보일지 확인. 내부 정보(일정·담당자
  개인명·지표)는 외부 뷰에 새면 안 됨
- **가드 우선**: 데이터를 덮어쓰는 코드는 "결과가 비었으면 저장 스킵", "일부 실패면
  이전 값 유지"를 기본 장착 ('26.7 실사고 2건의 교훈)
- **디자인 규칙 (위반 = 재작업)**: Pretendard · 흑백+현대그린 #0B4336 1색 · 형광 #EDFF5B는
  강조·매치 전용 · 빨강은 금지사항·경고에만 · 이모지 아이콘/box-shadow 카드/그라데이션/
  border-left 스트립 금지 · 표는 가로선만 · 위계는 타이포로
- **네이티브 prompt/confirm 금지** (환경별 차단) — 2단계 버튼 확인 패턴 사용
- **Enter 처리는 `!e.nativeEvent.isComposing` 체크** (한글 IME)

## 3. 구현 순서

데이터 계층(store/parse/데이터 파일) → UI(컴포넌트) → CSS(index.css, 기존 변수 사용)
→ 파서·데이터를 건드렸으면 **테스트 케이스 추가**(scripts/test-parse.mjs·test-data.mjs)
→ CLAUDE.md에 기능 한 줄 기록 ('26.7 형식 참고)

## 4. 검증 (건너뛰면 배포 사고)

1. `npm run verify` — 테스트+빌드. 실패 시 push 금지
2. UI를 바꿨으면 `npm run smoke` — 브라우저 자동 확인 (config 복원까지 자동)
3. 검증할 수 없는 부분(Supabase 실연동 등)은 **정직하게 "확인 못 함"으로 보고**

## 5. 배포와 보고

- 작업 브랜치 커밋(한국어, feat:/fix:) → push → main merge → push (= Vercel 자동 배포)

  사용자 액션(SQL 등)은 클릭 수준으로 분리 표기 → **후속 질문 3개**로 마무리
